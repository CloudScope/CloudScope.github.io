Curated Q&A for DevOps, Kubernetes, Docker, AWS, and Terraform interviews.

☸ Kubernetes

A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network namespace (same IP address and port space) and can share storage volumes. Containers within a Pod communicate over localhost. Pods are ephemeral — if they die, Kubernetes creates a new one rather than restarting the old one.

Both manage a set of pods, but they handle identity, storage, and ordering very differently.

Deployment — for stateless applications:

  • Pods are identical and interchangeable — no stable identity
  • Pods are named randomly (e.g. app-7d9f8b-xkq2p)
  • Can be scaled up/down and replaced in any order
  • All pods share the same PVC or use no persistent storage
  • Rolling updates replace pods freely without ordering guarantees
kubectl create deployment nginx --image=nginx --replicas=3

Use for: web servers, API services, microservices, workers — anything where every pod does the same job and data lives elsewhere.

StatefulSet — for stateful applications:

  • Each pod gets a stable, unique identity that persists across rescheduling (e.g. mysql-0, mysql-1, mysql-2)
  • Each pod gets its own PersistentVolumeClaim — storage is not shared and survives pod deletion
  • Pods start and stop in strict order (0 → 1 → 2 up, 2 → 1 → 0 down)
  • Each pod gets a stable DNS hostname: mysql-0.mysql.default.svc.cluster.local
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:          # ← each pod gets its own PVC
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi

Use for: databases (MySQL, PostgreSQL, MongoDB), message queues (Kafka, RabbitMQ), distributed systems (Elasticsearch, ZooKeeper) — anything where pod identity and per-pod storage matter.

A ConfigMap stores non-sensitive configuration data as key-value pairs (e.g. app settings, feature flags). A Secret stores sensitive data (passwords, tokens, TLS certificates) and values are base64-encoded. Secrets can be further protected with encryption at rest and RBAC policies. Always use Secrets for credentials, never ConfigMaps.
When you update a Deployment (e.g. new image tag), the Deployment controller creates a new ReplicaSet and gradually shifts pods from the old set to the new one. The pace is controlled by maxSurge (how many extra pods can exist) and maxUnavailable (how many can be down simultaneously). This ensures zero-downtime deploys. You can roll back with kubectl rollout undo deployment/name.
A Namespace is a virtual cluster within a Kubernetes cluster, used for resource isolation across teams, projects, or environments. Resources in different namespaces are isolated by default. You can apply ResourceQuotas and LimitRanges per namespace. Common defaults are default, kube-system, and kube-public.

You need a PersistentVolume (PV) with accessModes: [ReadWriteMany] (RWX), backed by a storage system that supports concurrent multi-node access — such as NFS, Amazon EFS, CephFS, Azure Files, or GlusterFS. Standard block storage (EBS, GCE PD) only supports ReadWriteOnce (single node).

Step 1 — Create a PersistentVolume (or use a StorageClass that provisions one):

apiVersion: v1
kind: PersistentVolume
metadata:
  name: shared-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteMany        # ← key: allows multiple nodes
  nfs:
    server: nfs-server.example.com
    path: /shared/data

Step 2 — Create a PersistentVolumeClaim:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 10Gi

Step 3 — Mount the same PVC in multiple pods:

apiVersion: v1
kind: Pod
metadata:
  name: pod-a
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - mountPath: /data
      name: shared-storage
  volumes:
  - name: shared-storage
    persistentVolumeClaim:
      claimName: shared-pvc   # ← same PVC name in pod-b too

Any number of pods — across different nodes — referencing the same claimName will all read and write to the same underlying storage. Use Amazon EFS on AWS or Azure Files on Azure for fully managed RWX storage without running your own NFS server.

🐳 Docker

ENTRYPOINT defines the executable that always runs when the container starts. CMD provides default arguments to that executable. CMD can be overridden at runtime (docker run image arg), but ENTRYPOINT requires --entrypoint to override. Best practice: use ENTRYPOINT for the main process and CMD for defaults.
A multi-stage build uses multiple FROM statements in one Dockerfile. Earlier stages compile or build the application; the final stage copies only the built artifacts into a minimal base image. This eliminates build tools, compilers, and intermediate files from the final image — dramatically reducing image size and attack surface. Example: build in golang:1.22, run in scratch or alpine.
COPY simply copies files or directories from the build context into the image. ADD does everything COPY does, plus it can auto-extract local tar archives and fetch remote URLs. Prefer COPY unless you specifically need auto-extraction, since ADD's extra behaviour can be surprising and harder to reason about.
Docker Compose lets you define and run multi-container applications using a single docker-compose.yml file. You declare each service, its image, environment variables, ports, volumes, and network dependencies. Running docker compose up starts the entire stack. It's ideal for local development and testing environments where you need multiple services (app + database + cache) to work together.
Key strategies: (1) Use a minimal base image like alpine or distroless. (2) Use multi-stage builds to exclude build tools. (3) Combine RUN commands to reduce layers. (4) Add a .dockerignore to exclude node_modules, .git, test files. (5) Remove caches in the same RUN layer: apt-get clean && rm -rf /var/lib/apt/lists/*. (6) Avoid installing unnecessary packages.

☁ AWS

EC2 is a virtual machine — you provision it, it runs continuously, and you're billed by the hour/second regardless of actual usage. Lambda is serverless and event-driven — code runs only when triggered, scales automatically to zero, and is billed per invocation and duration. Use EC2 for long-running workloads with predictable traffic; Lambda for short-lived event-driven tasks.
A Virtual Private Cloud (VPC) is an isolated network in AWS. Core components: Subnets (public for internet-facing, private for internal resources), Route Tables (control traffic routing), Internet Gateway (enables public subnet internet access), NAT Gateway (lets private subnets reach internet without being publicly reachable), Security Groups (stateful instance-level firewall), and NACLs (stateless subnet-level firewall).
Amazon S3 (Simple Storage Service) is object storage for any amount of data. Storage classes let you optimize cost vs. access frequency: Standard (frequent access), Standard-IA (infrequent access, lower cost), One Zone-IA (single AZ, cheaper), Glacier Instant Retrieval / Glacier Flexible Retrieval / Glacier Deep Archive (archival, cheapest), and Intelligent-Tiering (auto-moves objects between tiers based on access patterns).
AWS IAM (Identity and Access Management) controls who can do what in your AWS account. Best practices: (1) Apply least privilege — grant only the permissions needed. (2) Use IAM Roles for EC2/Lambda instead of embedding access keys. (3) Enable MFA for all human users. (4) Rotate access keys regularly. (5) Use IAM Groups to manage permissions at scale. (6) Never use the root account for day-to-day operations.

🏗 Terraform

Terraform state is a file (terraform.tfstate) that records the current state of your managed infrastructure. Terraform uses it to map real resources to your configuration, track metadata, and determine what changes are needed on the next apply. For teams, store state remotely (e.g. S3 + DynamoDB for locking) to enable collaboration and prevent concurrent modifications.
terraform plan is a dry run — it compares your configuration with the current state and shows exactly what will be created, updated, or destroyed without making any changes. terraform apply executes those changes after prompting for confirmation (or with -auto-approve in CI). Always run plan first in production pipelines and review the diff carefully.
Modules are reusable, parameterized groups of Terraform resources packaged together. A module takes input variables and produces outputs, encapsulating complexity. You can call the same module with different variables for dev/staging/prod environments. Modules can be local (relative path) or remote (Terraform Registry, Git repo). They promote DRY infrastructure code and consistent patterns across teams.
A data source allows Terraform to read information from existing infrastructure that it doesn't manage. For example, you can use data "aws_ami" to look up the latest Amazon Linux AMI ID without hardcoding it, or data "aws_vpc" to reference a VPC created outside your Terraform config. Data sources are declared with the data block and referenced as data.type.name.attribute.
