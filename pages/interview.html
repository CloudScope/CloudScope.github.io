☸ Kubernetes

A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network namespace (same IP address and port space) and can share storage volumes. Containers within a Pod communicate over localhost. Pods are ephemeral — if they die, Kubernetes creates a new one rather than restarting the old one.

Both manage a set of pods, but they handle identity, storage, and ordering very differently.

Deployment — for stateless applications:

  • Pods are identical and interchangeable — no stable identity
  • Pods are named randomly (e.g. app-7d9f8b-xkq2p)
  • Can be scaled up/down and replaced in any order
  • All pods share the same PVC or use no persistent storage
  • Rolling updates replace pods freely without ordering guarantees
kubectl create deployment nginx --image=nginx --replicas=3

Use for: web servers, API services, microservices, workers — anything where every pod does the same job and data lives elsewhere.

StatefulSet — for stateful applications:

  • Each pod gets a stable, unique identity that persists across rescheduling (e.g. mysql-0, mysql-1, mysql-2)
  • Each pod gets its own PersistentVolumeClaim — storage is not shared and survives pod deletion
  • Pods start and stop in strict order (0 → 1 → 2 up, 2 → 1 → 0 down)
  • Each pod gets a stable DNS hostname: mysql-0.mysql.default.svc.cluster.local
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: mysql
spec:
  serviceName: "mysql"
  replicas: 3
  selector:
    matchLabels:
      app: mysql
  template:
    metadata:
      labels:
        app: mysql
    spec:
      containers:
      - name: mysql
        image: mysql:8.0
        volumeMounts:
        - name: data
          mountPath: /var/lib/mysql
  volumeClaimTemplates:          # ← each pod gets its own PVC
  - metadata:
      name: data
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 10Gi

Use for: databases (MySQL, PostgreSQL, MongoDB), message queues (Kafka, RabbitMQ), distributed systems (Elasticsearch, ZooKeeper) — anything where pod identity and per-pod storage matter.

Quick comparison:

FeatureDeploymentStatefulSet
Pod identityRandom, replaceableStable, ordered
StorageShared or nonePer-pod PVC
Scale orderAny orderSequential
DNS hostnameRandomStable, predictable
Use caseStateless appsStateful apps / DBs
A ConfigMap stores non-sensitive configuration data as key-value pairs (e.g. app settings, feature flags). A Secret stores sensitive data (passwords, tokens, TLS certificates) and values are base64-encoded. Secrets can be further protected with encryption at rest and RBAC policies. Always use Secrets for credentials, never ConfigMaps.
When you update a Deployment (e.g. new image tag), the Deployment controller creates a new ReplicaSet and gradually shifts pods from the old set to the new one. The pace is controlled by maxSurge (how many extra pods can exist) and maxUnavailable (how many can be down simultaneously). This ensures zero-downtime deploys. You can roll back with kubectl rollout undo deployment/name.

A rolling update can silently drop requests if you don't configure the pod shutdown sequence carefully. There are four things you need to get right:

1. Readiness probe — only send traffic to ready pods

Kubernetes removes a pod from Service endpoints only when its readiness probe fails. New pods won't receive traffic until they pass. Without this, traffic hits pods that aren't ready yet.

readinessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5

2. maxUnavailable: 0 — never terminate an old pod before a new one is ready

strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1          # spin up 1 extra pod first
    maxUnavailable: 0    # never go below desired replica count

3. preStop hook + sleep — let kube-proxy drain the pod from iptables

When a pod is terminated, Kubernetes removes it from the Service endpoints list and sends SIGTERM at the same time. kube-proxy takes a few seconds to propagate the iptables rule change. Without a preStop sleep, new requests can still be routed to the dying pod during that gap.

lifecycle:
  preStop:
    exec:
      command: ["sleep", "5"]   # wait for kube-proxy to drain

4. terminationGracePeriodSeconds — allow in-flight requests to finish

After SIGTERM, give the app time to finish handling open connections before Kubernetes force-kills it with SIGKILL.

terminationGracePeriodSeconds: 30   # default is 30s; tune to your p99 latency

Full safe rollout config:

spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:
    spec:
      terminationGracePeriodSeconds: 30
      containers:
      - name: app
        image: myapp:v2
        lifecycle:
          preStop:
            exec:
              command: ["sleep", "5"]
        readinessProbe:
          httpGet:
            path: /healthz
            port: 8080
          periodSeconds: 5

With this setup: new pod passes readiness → old pod gets preStop sleep → kube-proxy drains → SIGTERM sent → app finishes in-flight requests → SIGKILL only after grace period. Zero dropped requests.

Kubernetes supports several strategies. The right choice depends on your risk tolerance, traffic volume, and rollback speed requirements.

1. Rolling Update (Kubernetes default)

Gradually replaces old pods with new ones. Zero downtime. Controlled by maxSurge and maxUnavailable.

strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # one extra pod during rollout
    maxUnavailable: 0  # never reduce below desired count

Use when: standard production deployments where both versions can run simultaneously and the API is backward compatible.

2. Recreate

Terminates ALL old pods, then creates new ones. Causes downtime between versions.

strategy:
  type: Recreate

Use when: v1 and v2 cannot run simultaneously (e.g., exclusive database schema locks, singleton processes). Accept the brief downtime.

3. Blue-Green (outside Kubernetes core — via Services or Ingress)

Run v1 (blue) and v2 (green) simultaneously. Switch all traffic at once by updating the Service selector or Ingress rule. Zero downtime; instant rollback by switching back.

# Switch Service from blue to green:
kubectl patch service my-app \
  -p '{"spec":{"selector":{"version":"green"}}}'

Use when: you need instant cutover and fast rollback, and can afford double infrastructure cost during the switch.

4. Canary

Route a small percentage of traffic to the new version. Increase gradually if metrics look healthy. Typically implemented with two Deployments and weighted Ingress rules (NGINX, Istio, or Argo Rollouts).

# With NGINX Ingress — send 10% to canary:
annotations:
  nginx.ingress.kubernetes.io/canary: "true"
  nginx.ingress.kubernetes.io/canary-weight: "10"

Use when: high-risk changes to critical services where you want to validate with real traffic before full rollout.

My default approach: Rolling Update with maxUnavailable: 0 + readiness probes + preStop sleep for most services. Canary for high-risk releases. Blue-Green when I need instant rollback with no traffic mixing.

Use Taints on nodes to repel pods, and Tolerations on pods to explicitly allow them onto tainted nodes. For node upgrades, combine this with PodDisruptionBudgets for graceful handling.

Taint a node (mark it as "off limits"):

# Syntax: key=value:effect
kubectl taint nodes node-1 upgrade=in-progress:NoSchedule

# Three taint effects:
# NoSchedule       — new pods won't be scheduled here (existing pods stay)
# PreferNoSchedule — scheduler avoids this node but won't guarantee it
# NoExecute        — new pods won't schedule AND existing pods are evicted

During a node upgrade — use NoExecute to gracefully drain:

# This evicts all pods that don't tolerate it (respects PDBs):
kubectl taint nodes node-1 upgrade=in-progress:NoExecute

# Equivalent to:
kubectl cordon node-1  # stop new scheduling
kubectl drain node-1 --ignore-daemonsets --delete-emptydir-data

Toleration on a pod (allow it to run on tainted nodes):

spec:
  tolerations:
  - key: "upgrade"
    operator: "Equal"
    value: "in-progress"
    effect: "NoSchedule"    # must match the taint effect

Common built-in taints you should know:

TaintWhen applied
node.kubernetes.io/not-readyNode is not healthy
node.kubernetes.io/unreachableNode controller can't reach the node
node.kubernetes.io/disk-pressureDisk is running low
node.kubernetes.io/memory-pressureMemory is running low

PodDisruptionBudget (PDB) ensures the drain respects your availability requirements — even with NoExecute, K8s won't evict a pod if doing so would violate the PDB:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: app-pdb
spec:
  minAvailable: 2   # always keep at least 2 pods up during eviction
  selector:
    matchLabels:
      app: my-app

Three things need to be right: networking, credentials, and connection management.

1. Networking — open the right security group

RDS must allow inbound traffic from your pods. The simplest and most common approach on EKS: allow the node group's security group to reach RDS on port 5432 (Postgres) or 3306 (MySQL).

# Terraform: allow EKS nodes to reach RDS
resource "aws_security_group_rule" "eks_to_rds" {
  type                     = "ingress"
  from_port                = 5432
  to_port                  = 5432
  protocol                 = "tcp"
  security_group_id        = aws_security_group.rds.id
  source_security_group_id = aws_security_group.eks_nodes.id
}

2. Credentials — never hardcode; use Secrets

Option A: Kubernetes Secret (simple, acceptable for non-production)

kubectl create secret generic rds-creds \
  --from-literal=DB_HOST=mydb.xxxxx.us-east-1.rds.amazonaws.com \
  --from-literal=DB_USER=app \
  --from-literal=DB_PASSWORD=supersecret
# Inject into pod as env vars:
envFrom:
- secretRef:
    name: rds-creds

Option B: AWS Secrets Manager + Secrets Store CSI Driver (recommended for production)

Store credentials in Secrets Manager, mount them as files or env vars at pod startup — no secret stored in etcd.

volumes:
- name: rds-secret
  csi:
    driver: secrets-store.csi.k8s.io
    readOnly: true
    volumeAttributes:
      secretProviderClass: rds-credentials

Option C: IAM authentication for RDS (most secure, passwordless)

Enable IAM DB authentication on the RDS instance, use IRSA (IAM Roles for Service Accounts) to give the pod an IAM role, and generate a short-lived token at connection time instead of using a password.

# IRSA annotation on service account:
annotations:
  eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/app-rds-role

3. Connection management

  • Always use a connection pool (PgBouncer, HikariCP) — RDS has per-instance connection limits
  • Connect to the RDS endpoint hostname, not IP — it changes on failover
  • Enable SSL: sslmode=require (Postgres) or requireSSL=true (MySQL)
  • Use the reader endpoint for read replicas to offload read traffic

This depends on how the ConfigMap is consumed. The behaviour is different for env vars vs volume mounts.

Env vars (envFrom / env.valueFrom) — do NOT auto-update

Environment variables are set at container start time and never change while the pod is running, even if the ConfigMap changes. The pod must be restarted to pick up the new value.

Volume mounts — DO auto-update (eventually)

When a ConfigMap is mounted as a file, the kubelet syncs changes roughly every 60 seconds. The file on disk gets updated without a pod restart. This works if your application watches the file for changes (e.g., Nginx, many config-aware apps).

The Helm trick for zero-downtime rolling restart on ConfigMap change

Helm can't detect that a ConfigMap changed and trigger a rolling update on its own — unless you add a checksum annotation to the Deployment pod template. When the ConfigMap content changes, the SHA256 changes, the pod template hash changes, and Kubernetes performs a rolling update automatically:

# In your Helm chart — templates/deployment.yaml:
spec:
  template:
    metadata:
      annotations:
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}

Now helm upgrade on a ConfigMap change will trigger a zero-downtime rolling restart of the Deployment using the normal Rolling Update strategy (maxSurge, maxUnavailable, readiness probes).

Alternative: use kubectl rollout restart in your CI pipeline

helm upgrade my-app ./chart --values prod-values.yaml
kubectl rollout restart deployment/my-app

Best practice for production: use the checksum/config annotation so restarts are tied to actual config changes, not manual steps — and always pair with maxUnavailable: 0 and readiness probes to guarantee zero dropped requests during the restart.

By default Kubernetes may schedule multiple pods from the same Deployment on the same node. There are two ways to prevent this.

Option 1 — Pod Anti-Affinity (classic approach)

Use podAntiAffinity to tell the scheduler "don't place me on a node that already has a pod with this label."

spec:
  template:
    metadata:
      labels:
        app: my-app
    spec:
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:  # hard rule
          - labelSelector:
              matchLabels:
                app: my-app
            topologyKey: "kubernetes.io/hostname"  # one pod per node

Use preferredDuringSchedulingIgnoredDuringExecution instead for a soft preference — useful when you have more replicas than nodes and don't want scheduling to fail.

Option 2 — TopologySpreadConstraints (recommended, more flexible)

Introduced in Kubernetes 1.18, this gives fine-grained control over how pods are distributed across nodes, zones, or any topology key.

spec:
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1                          # max difference in pod count between nodes
        topologyKey: kubernetes.io/hostname
        whenUnsatisfiable: DoNotSchedule    # hard: fail if can't spread
        labelSelector:
          matchLabels:
            app: my-app

maxSkew: 1 means the busiest node can have at most 1 more pod than the least busy. Set whenUnsatisfiable: ScheduleAnyway for a best-effort spread without blocking scheduling.

Which to use? Prefer topologySpreadConstraints — it's more expressive, handles multi-zone spreading, and is the Kubernetes-recommended approach going forward. Use anti-affinity when you need strict "exactly one pod per node" guarantees.

A Namespace is a virtual cluster within a Kubernetes cluster, used for resource isolation across teams, projects, or environments. Resources in different namespaces are isolated by default. You can apply ResourceQuotas and LimitRanges per namespace. Common defaults are default, kube-system, and kube-public.

EKS upgrades happen in two stages: control plane then data plane. Both can be done with zero downtime if you follow the right order.

Stage 1 — Upgrade the control plane (API server, etcd, scheduler)

This is fully managed by AWS and takes ~10 minutes. The API server may be briefly unavailable during the upgrade but running pods are unaffected — the data plane keeps serving traffic independently.

# Via AWS Console, CLI, or Terraform:
aws eks update-cluster-version \
  --name my-cluster \
  --kubernetes-version 1.30

# Watch progress:
aws eks describe-update --name my-cluster --update-id <id>

Stage 2 — Upgrade add-ons to match new control plane version

Before touching nodes, update managed add-ons: kube-proxy, CoreDNS, vpc-cni. Version mismatches can cause networking issues.

aws eks update-addon \
  --cluster-name my-cluster \
  --addon-name kube-proxy \
  --addon-version v1.30.0-eksbuild.3

Stage 3 — Set PodDisruptionBudgets before touching nodes

PDBs prevent the node drainer from evicting too many pods at once, ensuring your service stays available during the node replacement:

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: app-pdb
spec:
  minAvailable: 2       # always keep at least 2 pods running
  selector:
    matchLabels:
      app: my-app

Stage 4 — Roll node groups one at a time

For managed node groups, trigger a rolling update. EKS drains each node (respecting PDBs), terminates it, launches a replacement, and waits for it to be Ready before moving to the next.

# Managed node group — rolling update:
aws eks update-nodegroup-version \
  --cluster-name my-cluster \
  --nodegroup-name workers \
  --kubernetes-version 1.30

# Self-managed nodes — cordon + drain manually:
kubectl cordon node-1
kubectl drain node-1 --ignore-daemonsets --delete-emptydir-data

Key rules:

  • Always upgrade control plane before nodes
  • Only skip one minor version at a time (1.28 → 1.29 → 1.30)
  • Upgrade add-ons before nodes, not after
  • PDBs are your safety net — set them before starting
  • Test the upgrade in staging on the same minor version first

You need a PersistentVolume (PV) with accessModes: [ReadWriteMany] (RWX), backed by a storage system that supports concurrent multi-node access — such as NFS, Amazon EFS, CephFS, Azure Files, or GlusterFS. Standard block storage (EBS, GCE PD) only supports ReadWriteOnce (single node).

Step 1 — Create a PersistentVolume (or use a StorageClass that provisions one):

apiVersion: v1
kind: PersistentVolume
metadata:
  name: shared-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteMany        # ← key: allows multiple nodes
  nfs:
    server: nfs-server.example.com
    path: /shared/data

Step 2 — Create a PersistentVolumeClaim:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: shared-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 10Gi

Step 3 — Mount the same PVC in multiple pods:

apiVersion: v1
kind: Pod
metadata:
  name: pod-a
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - mountPath: /data
      name: shared-storage
  volumes:
  - name: shared-storage
    persistentVolumeClaim:
      claimName: shared-pvc   # ← same PVC name in pod-b too

Any number of pods — across different nodes — referencing the same claimName will all read and write to the same underlying storage. Use Amazon EFS on AWS or Azure Files on Azure for fully managed RWX storage without running your own NFS server.

By default — one ALB per Ingress. Each Ingress resource the AWS Load Balancer Controller (formerly ALB Ingress Controller) sees gets its own ALB. For large clusters with many services this gets expensive quickly.

With IngressGroup — multiple Ingresses share one ALB. Add the same group.name annotation to all Ingresses you want to merge:

# service-a ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: service-a-ingress
  annotations:
    kubernetes.io/ingress.class: alb
    alb.ingress.kubernetes.io/group.name: shared-alb    # ← group key
    alb.ingress.kubernetes.io/group.order: "10"         # rule priority
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /users
        pathType: Prefix
        backend:
          service:
            name: service-a
            port:
              number: 80
# service-b ingress — same group.name
metadata:
  name: service-b-ingress
  annotations:
    alb.ingress.kubernetes.io/group.name: shared-alb
    alb.ingress.kubernetes.io/group.order: "20"
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /orders
        pathType: Prefix
        backend:
          service:
            name: service-b
            port:
              number: 80

The controller merges listener rules from all Ingresses in the group onto a single ALB. group.order controls which rules take priority when paths overlap (lower number = higher priority).

Things to be aware of:

  • All services in a group share the same ALB security group — WAF rules, access logs, and SSL cert settings apply to all
  • Deleting one Ingress only removes its rules from the shared ALB — the ALB itself stays up
  • The ALB is only deleted when the last Ingress in the group is deleted
  • Use IngressGroup when services are in the same security domain; use separate ALBs to isolate them

High restart counts usually mean one of four things: OOMKilled, app crash on startup, liveness probe failure, or a missing config/secret. Here's the debugging sequence:

Step 1 — Check restart count and current status

kubectl get pods -n <namespace>
# Look at RESTARTS column and STATUS
# CrashLoopBackOff = container keeps dying and K8s is backing off retries

Step 2 — Read the last container state

kubectl describe pod <pod-name> -n <namespace>
# Key fields to look at:
#   Last State: Terminated
#   Reason: OOMKilled / Error / Completed
#   Exit Code: 137 / 1 / 0
#   Events section at the bottom

Step 3 — Read logs from the crashed container

kubectl logs <pod-name> --previous   # logs from the container that just died
kubectl logs <pod-name> -f           # live logs from current container

Decode the Exit Code:

Exit CodeMeaningFix
137OOMKilled — container exceeded memory limitIncrease resources.limits.memory or fix memory leak
1App crashed — unhandled error on startupCheck --previous logs for stack trace
0Clean exit — app exited itselfApp shouldn't exit in a pod; check entrypoint
143SIGTERM received — liveness probe killed itTune probe failureThreshold and periodSeconds

Common causes and fixes:

  • OOMKilled — container hit memory limit. Check actual usage: kubectl top pod <name>, then raise the limit or profile for leaks
  • CrashLoopBackOff — app fails immediately on start. Usually a bad env var, missing secret mount, or startup bug. Check --previous logs
  • Liveness probe too aggressive — probe fires before app is ready. Increase initialDelaySeconds or failureThreshold
  • Missing ConfigMap/Secret — pod can't mount volume. Check describe pod Events for MountVolume failed

🐳 Docker

ENTRYPOINT defines the executable that always runs when the container starts. CMD provides default arguments to that executable. CMD can be overridden at runtime (docker run image arg), but ENTRYPOINT requires --entrypoint to override. Best practice: use ENTRYPOINT for the main process and CMD for defaults.
A multi-stage build uses multiple FROM statements in one Dockerfile. Earlier stages compile or build the application; the final stage copies only the built artifacts into a minimal base image. This eliminates build tools, compilers, and intermediate files from the final image — dramatically reducing image size and attack surface. Example: build in golang:1.22, run in scratch or alpine.
COPY simply copies files or directories from the build context into the image. ADD does everything COPY does, plus it can auto-extract local tar archives and fetch remote URLs. Prefer COPY unless you specifically need auto-extraction, since ADD's extra behaviour can be surprising and harder to reason about.
Docker Compose lets you define and run multi-container applications using a single docker-compose.yml file. You declare each service, its image, environment variables, ports, volumes, and network dependencies. Running docker compose up starts the entire stack. It's ideal for local development and testing environments where you need multiple services (app + database + cache) to work together.
Key strategies: (1) Use a minimal base image like alpine or distroless. (2) Use multi-stage builds to exclude build tools. (3) Combine RUN commands to reduce layers. (4) Add a .dockerignore to exclude node_modules, .git, test files. (5) Remove caches in the same RUN layer: apt-get clean && rm -rf /var/lib/apt/lists/*. (6) Avoid installing unnecessary packages.

☁ AWS

EC2 is a virtual machine — you provision it, it runs continuously, and you're billed by the hour/second regardless of actual usage. Lambda is serverless and event-driven — code runs only when triggered, scales automatically to zero, and is billed per invocation and duration. Use EC2 for long-running workloads with predictable traffic; Lambda for short-lived event-driven tasks.
A Virtual Private Cloud (VPC) is an isolated network in AWS. Core components: Subnets (public for internet-facing, private for internal resources), Route Tables (control traffic routing), Internet Gateway (enables public subnet internet access), NAT Gateway (lets private subnets reach internet without being publicly reachable), Security Groups (stateful instance-level firewall), and NACLs (stateless subnet-level firewall).
Amazon S3 (Simple Storage Service) is object storage for any amount of data. Storage classes let you optimize cost vs. access frequency: Standard (frequent access), Standard-IA (infrequent access, lower cost), One Zone-IA (single AZ, cheaper), Glacier Instant Retrieval / Glacier Flexible Retrieval / Glacier Deep Archive (archival, cheapest), and Intelligent-Tiering (auto-moves objects between tiers based on access patterns).
AWS IAM (Identity and Access Management) controls who can do what in your AWS account. Best practices: (1) Apply least privilege — grant only the permissions needed. (2) Use IAM Roles for EC2/Lambda instead of embedding access keys. (3) Enable MFA for all human users. (4) Rotate access keys regularly. (5) Use IAM Groups to manage permissions at scale. (6) Never use the root account for day-to-day operations.

An ECS Task is a single running instance of a task definition — think of it as one docker run. It runs, does its job, and stops. ECS doesn't restart it automatically.

An ECS Service is a long-running controller that ensures a desired number of Tasks are always running. If a Task crashes, the Service replaces it. It integrates with ALB/NLB for load balancing and handles rolling deployments.

FeatureECS Task (standalone)ECS Service
LifecycleRuns once, then stopsAlways keeps N tasks running
Auto-restart on failureNoYes
Load balancer integrationNoYes (ALB/NLB)
Rolling deploymentsNoYes
Use caseBatch jobs, DB migrations, cron (via EventBridge)Web APIs, background workers, long-running apps

The Kubernetes analogy: a standalone ECS Task ≈ a Kubernetes Job; an ECS Service ≈ a Kubernetes Deployment.

These are two distinct IAM roles that serve completely different purposes and are a common source of confusion.

Task Execution Role — used by the ECS agent (infrastructure), not your application code:

  • Pull container images from ECR
  • Write container logs to CloudWatch Logs
  • Retrieve secrets from AWS Secrets Manager or SSM Parameter Store to inject as environment variables
# Typical execution role policy:
{
  "Effect": "Allow",
  "Action": [
    "ecr:GetAuthorizationToken",
    "ecr:BatchPullImage",
    "logs:CreateLogStream",
    "logs:PutLogEvents",
    "secretsmanager:GetSecretValue"
  ],
  "Resource": "*"
}

Task Role — used by your application code running inside the container:

  • Read/write S3 buckets
  • Query DynamoDB tables
  • Send messages to SQS/SNS
  • Any AWS SDK calls your app makes at runtime
# Typical task role policy (least-privilege example):
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::my-app-bucket/*"
}

Memory aid: Execution Role = "ECS needs this to run your container." Task Role = "Your app needs this to do its job."

A common mistake is putting application permissions in the Execution Role — they end up in the same place but semantically they're wrong, and it violates least-privilege. Always split them.

Node Groups are the managed compute resources (EC2 instances) that run your containerized workloads in EKS. They are groups of identical EC2 worker nodes managed together as a unit.

EKS offers two types:

Managed Node Groups (recommended)

  • AWS provisions, registers, and manages the EC2 instances for you
  • Automatic, graceful node replacements during upgrades (drains pods, respects PDBs)
  • Integrated with EC2 Auto Scaling Groups under the hood
  • Nodes automatically register with the cluster — no bootstrap scripts needed
# Terraform example:
resource "aws_eks_node_group" "workers" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "workers"
  node_role_arn   = aws_iam_role.node.arn
  subnet_ids      = var.private_subnet_ids

  instance_types = ["m5.large"]
  scaling_config {
    desired_size = 3
    min_size     = 2
    max_size     = 10
  }
}

Self-Managed Node Groups

  • You manage the EC2 instances, Auto Scaling Groups, and bootstrap configuration yourself
  • More control (custom AMIs, custom kubelet flags) but more operational overhead
  • Use when you need a specific OS, GPU instances, or non-standard configurations

Fargate (no node groups) — serverless option where AWS runs pods directly without you managing EC2 nodes at all. No visibility into the underlying instance.

Common pattern: Use multiple node groups with different instance types — a general-purpose group for standard workloads and a GPU or memory-optimized group for specific jobs, controlled via node selectors or taints.

🏗 Terraform

Terraform state is a file (terraform.tfstate) that records the current state of your managed infrastructure. Terraform uses it to map real resources to your configuration, track metadata, and determine what changes are needed on the next apply. For teams, store state remotely (e.g. S3 + DynamoDB for locking) to enable collaboration and prevent concurrent modifications.
terraform plan is a dry run — it compares your configuration with the current state and shows exactly what will be created, updated, or destroyed without making any changes. terraform apply executes those changes after prompting for confirmation (or with -auto-approve in CI). Always run plan first in production pipelines and review the diff carefully.
Modules are reusable, parameterized groups of Terraform resources packaged together. A module takes input variables and produces outputs, encapsulating complexity. You can call the same module with different variables for dev/staging/prod environments. Modules can be local (relative path) or remote (Terraform Registry, Git repo). They promote DRY infrastructure code and consistent patterns across teams.
A data source allows Terraform to read information from existing infrastructure that it doesn't manage. For example, you can use data "aws_ami" to look up the latest Amazon Linux AMI ID without hardcoding it, or data "aws_vpc" to reference a VPC created outside your Terraform config. Data sources are declared with the data block and referenced as data.type.name.attribute.

This is a two-part problem: safely introducing a new cluster (us-east-2) and safely upgrading an existing one (us-east-1). The risks on the existing cluster are resource recreation, accidental destroy, and state drift. Here's how to handle each.

1. Use separate state per cluster — never share state between regions

Each cluster should have its own Terraform state file, either via workspaces or separate root module directories. This ensures changes to one cluster can never accidentally affect the other.

# Option A: separate directories
infra/
  eks-us-east-1/   # existing cluster
    main.tf
    backend.tf     # S3 key: eks/us-east-1/terraform.tfstate
  eks-us-east-2/   # new cluster
    main.tf
    backend.tf     # S3 key: eks/us-east-2/terraform.tfstate

2. Deploy us-east-2 first — it's greenfield, no risk

Apply the new module version to us-east-2 fresh. This validates the new module works correctly before you touch production.

cd infra/eks-us-east-2
terraform init
terraform plan   # review, then:
terraform apply

3. Before upgrading us-east-1 — run terraform plan and read it carefully

Module upgrades often rename resources or change argument defaults. plan will reveal any destructive changes. Watch for these warning signs:

# DANGEROUS — this will destroy your cluster:
- aws_eks_cluster.main  # forces replacement

# SAFE — in-place update:
~ aws_eks_cluster.main
    version: "1.28" -> "1.29"

4. Handle resource renames with terraform state mv

If the new module version renames a resource internally (e.g. aws_eks_cluster.eks → aws_eks_cluster.main), Terraform sees a destroy + create. Fix this by moving the resource in state — no real-world change happens:

terraform state mv \
  module.eks.aws_eks_cluster.eks \
  module.eks.aws_eks_cluster.main

Re-run terraform plan — the dangerous destroy should disappear.

5. Apply us-east-1 changes incrementally using -target

For high-risk upgrades, apply one resource at a time rather than everything at once:

terraform apply -target=module.eks.aws_eks_node_group.workers

6. Pin module versions explicitly

Never use an unpinned module source. This prevents unintended upgrades in CI:

module "eks" {
  source  = "terraform-aws-modules/eks/aws"
  version = "~> 20.0"   # ← always pin; review changelog before bumping
}

7. Check the module's CHANGELOG for breaking changes

Before upgrading, read the module's release notes. Major version bumps (v19 → v20) typically have a migration guide listing required state mv operations and removed variables.

Summary checklist:

  • Separate state per cluster/region
  • Deploy us-east-2 first to validate new module
  • Run terraform plan on us-east-1 — look for forces replacement
  • Use terraform state mv to resolve renames without recreation
  • Apply incrementally with -target for risky resources
  • Pin module versions; read CHANGELOG before bumping

🔄 CI/CD

This is a classic copy-paste-pipeline problem I've solved with reusable workflows. You define the pipeline once in a central repo and each service calls it, so a change is made in exactly one place instead of 15.

1. Define the reusable workflow with workflow_call

A reusable workflow is a normal workflow file that is triggered by on: workflow_call instead of push/pull_request. It declares a typed contract of inputs, secrets, and optional outputs. I'd put this in a dedicated org/ci-workflows repo under .github/workflows/:

# org/ci-workflows/.github/workflows/build-deploy.yml
name: build-deploy
on:
  workflow_call:
    inputs:
      service_name:
        required: true
        type: string
      ecr_repo:
        required: true
        type: string
      environment:
        required: false
        type: string
        default: staging
    secrets:
      AWS_ROLE_ARN:
        required: true
    outputs:
      image_tag:
        description: "Pushed image tag"
        value: ${{ jobs.build.outputs.image_tag }}

jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image_tag: ${{ steps.meta.outputs.tag }}
    permissions:
      id-token: write      # OIDC for AWS auth
      contents: read
    steps:
      - uses: actions/checkout@v4
      - id: meta
        run: echo "tag=${GITHUB_SHA::7}" >> "$GITHUB_OUTPUT"
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_ARN }}
          aws-region: us-east-1
      - name: Build, test, scan & push
        run: |
          docker build -t $ECR_REPO:${{ steps.meta.outputs.tag }} .
          trivy image --exit-code 1 $ECR_REPO:${{ steps.meta.outputs.tag }}
          docker push $ECR_REPO:${{ steps.meta.outputs.tag }}
        env:
          ECR_REPO: ${{ inputs.ecr_repo }}

2. Each service becomes a thin caller

Every one of the 15 services now has a tiny workflow that just calls the central one and passes its own parameters. The whole per-service file is ~10 lines instead of ~150:

# payments-service/.github/workflows/ci.yml
name: CI
on:
  push:
    branches: [main]

jobs:
  pipeline:
    uses: my-org/ci-workflows/.github/workflows/build-deploy.yml@v1
    with:
      service_name: payments
      ecr_repo: 1234.dkr.ecr.us-east-1.amazonaws.com/payments
      environment: production
    secrets:
      AWS_ROLE_ARN: ${{ secrets.AWS_ROLE_ARN }}
    permissions:
      id-token: write
      contents: read

Note the caller references the workflow with a ref — @v1. You can pin to a tag, branch, or full SHA. I pin to a tag (or SHA for security-sensitive orgs) so a change in the central repo doesn't silently break all 15 pipelines — I bump the tag deliberately after testing.

3. Passing secrets cleanly

Reusable workflows do not inherit the caller's secrets automatically. You either map each secret explicitly under secrets: (as above), or use secrets: inherit to forward all of the caller's secrets — handy when they live at the org level:

    uses: my-org/ci-workflows/.github/workflows/build-deploy.yml@v1
    with: { service_name: payments, ecr_repo: ... }
    secrets: inherit

4. Reusable workflow vs composite action — the key distinction

This is the follow-up interviewers always ask. Both promote reuse but operate at different levels:

  • Reusable workflow — reuses an entire workflow: one or more jobs, each on its own runner, with its own permissions, environment, strategy matrix, and job-level secrets. Called at the jobs.<id>.uses level. Best for standardizing a whole pipeline (build → deploy).
  • Composite action — reuses a sequence of steps inside a single job. Called at the steps.uses level. It can't define multiple jobs or job-level permissions. Best for a reusable chunk of steps like "setup + login + cache" that you drop into many jobs.

Rule of thumb: need multiple jobs / matrix / environments → reusable workflow; need to package a few steps → composite action.

5. Gotchas I mention from real experience

  • Nesting is limited — you can nest reusable workflows up to 4 levels deep.
  • A caller job that uses uses: can't also define its own steps: — the called workflow is the job body.
  • env defined at the caller level is not propagated into the reusable workflow — pass values via inputs instead.
  • Set the called workflow's visibility (private repos need "Access" enabled under Actions settings) or the callers get a "workflow not found" error.
  • Version the central repo with tags and a changelog; treat it like a shared library, because breaking it breaks every consumer.

Result: the 15 near-identical pipelines collapse into one source of truth plus 15 ten-line callers. A pipeline change is a single PR to ci-workflows, tested, tagged, then rolled out by bumping the ref — exactly the DRY, auditable setup you want at scale.

This trips up a lot of teams. A reusable workflow in a private repo is not shareable by default — GitHub blocks other repos from referencing it until you explicitly grant access. There's a single setting that fixes it, plus a few related gotchas.

1. Grant Actions access on the source repo (the fix)

On the org/ci-workflows repo, go to Settings → Actions → General → Access and set the access policy. The relevant option is:

  • "Accessible from repositories in the '<org>' organization" — lets any repo in your org call this repo's workflows and actions. This is the one you almost always want for a shared ci-workflows repo.
  • "Not accessible" (default) — only this repo itself can use its workflows.
  • "Accessible from repositories owned by the user/enterprise" — broader, for cross-org sharing on Enterprise.

You can also automate this via the REST API instead of clicking through the UI:

gh api -X PUT \
  /repos/my-org/ci-workflows/actions/permissions/access \
  -f access_level=organization

Once set, the existing uses: my-org/ci-workflows/.github/workflows/build-deploy.yml@v1 reference just works — no change needed in the callers.

2. Make sure the caller repo is allowed to run those actions

The other half is the consumer side. On each calling repo (or org-wide under Organization Settings → Actions → Policies), the allowed-actions policy must permit the workflows/actions you're pulling in. For an internal shared repo, "Allow <org>, and select non-<org>, actions" with the org's own actions enabled covers it.

3. Checkout of private code inside the workflow needs a token

Granting Actions access lets a repo call the workflow, but if that reusable workflow (or a step in it) needs to checkout a different private repo, the default GITHUB_TOKEN is scoped only to the caller repo. For that you need broader credentials:

      - uses: actions/checkout@v4
        with:
          repository: my-org/shared-helm-charts
          token: ${{ secrets.ORG_PAT }}   # PAT or GitHub App token
                                          # with repo:read on the target

Prefer a GitHub App installation token or a fine-grained PAT over a classic PAT, and store it as an org-level secret so you rotate it in one place.

4. Common "workflow not found" causes to rule out

  • Access policy still on "Not accessible" (step 1) — the #1 cause.
  • Wrong ref — the tag/branch/SHA in @v1 doesn't exist in ci-workflows.
  • Path typo — it must be the full .github/workflows/<file>.yml path, not just the filename.
  • The reusable workflow file isn't on the default branch and you referenced it by a ref that predates it.
  • SSO not authorized on the PAT (if using one) for the org.

Summary: flip the source repo's Actions Access policy to organization-scoped (one setting or one API call), confirm the caller's allowed-actions policy permits it, and use an App/PAT token only if the workflow additionally checks out other private repos.

⚙ General DevOps

A production-grade pipeline has seven stages. Here's how I'd design each one:

1. Source Control (Git)

All code — application, IaC, Helm charts, Dockerfiles — lives in Git (GitHub/GitLab). Branch strategy: main = production, staging, short-lived feature branches. PRs require at least one review + passing CI before merge.

2. CI — Build, Test & Code Quality

Triggered on every PR and push. Runs in parallel:

  • Unit tests + integration tests
  • Static code analysis (SonarQube, ESLint)
  • Dependency vulnerability scan (Dependabot, Snyk)
  • Docker image build
  • Container security scan (Trivy) — fail pipeline on HIGH/CRITICAL CVEs
# GitHub Actions example:
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: npm test
    - run: docker build -t myapp:${{ github.sha }} .
    - run: trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:${{ github.sha }}

3. Artifact Registry

Push the verified image to ECR (or Docker Hub) tagged with the Git SHA — never :latest in production. Immutable tags mean you always know exactly what's running.

docker tag myapp:$SHA $ECR_REPO:$SHA
docker push $ECR_REPO:$SHA

4. Infrastructure as Code (Terraform)

Separate pipeline for infra changes. terraform plan runs on every PR (output posted as a PR comment). terraform apply runs only after merge to main, with remote state in S3 + DynamoDB locking.

5. CD — Deploy to Kubernetes via Helm/Argo CD

Two approaches:

  • Push-based (CI-driven): CI pipeline runs helm upgrade directly after image push
  • Pull-based / GitOps (recommended): Argo CD or Flux watches the Git repo for changes to Helm values (image tag) and syncs the cluster automatically

Promotion order: Dev → Staging → Production. Production requires a manual approval gate.

# GitOps: update image tag in values file, commit, Argo CD picks it up
image:
  repository: 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp
  tag: "a3f9c21"   # ← updated by CI after image push

6. Observability

  • Metrics: Prometheus + Grafana (dashboards for latency, error rate, saturation)
  • Logs: FluentBit → CloudWatch Logs / Elasticsearch
  • Traces: AWS X-Ray or OpenTelemetry
  • Alerts: PagerDuty / Slack for SLO breaches (error rate > 1%, p99 latency > 500ms)

7. Rollback strategy

Every deploy is a Helm release — rollback is a one-liner: helm rollback my-app 3. With Argo CD/GitOps, rollback = revert the Git commit and the cluster self-heals.

Summary diagram:

Developer → Git PR → CI (test + build + scan) → ECR (image push)
     ↓
Argo CD detects new image tag → Rolling deploy to Dev
     ↓ (auto-promote if tests pass)
Staging → Manual approval gate → Production
