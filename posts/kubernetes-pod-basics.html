Kubernetes Pods: Everything You Need to Know

📅 Jul 10, 2025 ⏱ 8 min read ✍ Raghunath Pradhan

What is a Pod?

A Pod is the smallest and most fundamental deployable unit in Kubernetes — not a container, but a wrapper around one or more containers. All containers in a Pod share the same network namespace, meaning they all have the same IP address and can communicate with each other over localhost. They can also share storage volumes mounted into the Pod.

Pods are ephemeral by design. If a Pod dies, Kubernetes doesn't restart it — it creates a brand-new Pod. Higher-level abstractions like Deployments and StatefulSets manage Pods and ensure the desired number are always running.

Think of a Pod as a logical host — containers inside it behave as if they're running on the same machine.

Pod Anatomy

Here is a minimal Pod manifest:

apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
spec:
  containers:
    - name: nginx
      image: nginx:1.25-alpine
      ports:
        - containerPort: 80
      resources:
        requests:
          memory: "64Mi"
          cpu: "250m"
        limits:
          memory: "128Mi"
          cpu: "500m"

Key fields to know:

  • metadata.labels — used by Services and selectors to target this Pod
  • spec.containers[].image — the container image to run
  • resources.requests — minimum guaranteed resources for scheduling
  • resources.limits — maximum resources the container can consume

Pod Lifecycle

Every Pod moves through a defined set of phases:

PhaseMeaning
PendingPod accepted by the cluster; containers not yet started (e.g. image pulling or scheduling in progress)
RunningAt least one container is running, starting, or restarting
SucceededAll containers exited successfully (exit code 0) and will not restart
FailedAll containers have terminated; at least one exited with non-zero code
UnknownPod state cannot be determined — usually a node communication issue

Within Running, individual containers have their own state: Waiting, Running, or Terminated. The restartPolicy (Always, OnFailure, Never) controls what happens when a container exits.

Essential kubectl Commands

These commands cover the majority of day-to-day Pod operations:

# List all pods in the current namespace
kubectl get pods

# List pods across all namespaces
kubectl get pods -A

# Describe a specific pod (events, conditions, container states)
kubectl describe pod <pod-name>

# Stream logs from a pod
kubectl logs <pod-name>

# Stream logs from a specific container in a multi-container pod
kubectl logs <pod-name> -c <container-name>

# Open a shell inside a running pod
kubectl exec -it <pod-name> -- /bin/sh

# Delete a pod (the controller will recreate it)
kubectl delete pod <pod-name>

# Apply a pod manifest
kubectl apply -f pod.yaml

# Watch pod status in real time
kubectl get pods -w

Multi-Container Pod Patterns

While most Pods run a single container, multi-container Pods enable powerful patterns where helper containers augment the main application container.

Sidecar Pattern

A sidecar container extends the main container. Example: a log-shipping sidecar reads logs written to a shared volume by the app container and forwards them to a central logging service (Fluentd, Filebeat). The main app doesn't need to know anything about log shipping.

Ambassador Pattern

The ambassador container acts as a proxy, handling external communication on behalf of the main container. Example: a local proxy container routes database connections to the correct shard, so the app always talks to localhost:5432.

Adapter Pattern

The adapter standardizes and transforms output from the main container. Example: a container running a legacy app that exposes metrics in a proprietary format gets an adapter sidecar that converts those metrics to Prometheus format for scraping.

Key Interview Questions

Q: Can containers in the same Pod communicate without a Service?
Yes — they share the same network namespace, so they communicate via localhost and the target container's port.

Q: What happens to a Pod when its node fails?
If the node becomes unreachable, the Pod enters Unknown state. After a configurable timeout, the controller marks the Pod for eviction and reschedules it on a healthy node (for Deployments/ReplicaSets). StatefulSet Pods require manual intervention by default.

Q: What is the difference between a Pod restart and a Pod recreation?
A restart occurs when a container inside an existing Pod crashes and the kubelet restarts it (per restartPolicy). The Pod keeps the same name and IP. A recreation happens when the Pod itself is deleted and the controller (Deployment, ReplicaSet) creates an entirely new Pod with a new name and IP.
