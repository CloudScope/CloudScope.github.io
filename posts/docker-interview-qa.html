Top Docker Interview Questions & Answers (2025)

📅 Jul 8, 2025 ⏱ 12 min read ✍ Raghunath Pradhan

1. What is Docker and why use it?

Docker is an open-source platform that packages applications and their dependencies into lightweight, portable units called containers. Containers share the host OS kernel but are isolated using Linux namespaces and cgroups, making them faster and more resource-efficient than virtual machines.

The key benefit is environment consistency — a container runs identically on a developer's laptop, a CI server, and production. This eliminates the classic "works on my machine" problem and simplifies deployment pipelines significantly.

2. What is the difference between CMD and ENTRYPOINT?

ENTRYPOINT sets the executable that always runs when the container starts. CMD provides default arguments to that executable and can be overridden at runtime. 

# ENTRYPOINT sets the fixed executable
ENTRYPOINT ["python3", "app.py"]

# CMD provides overridable defaults
CMD ["--port", "8080"]

# At runtime, you can override CMD:
# docker run myimage --port 9090

# To override ENTRYPOINT you need --entrypoint:
# docker run --entrypoint /bin/sh myimage

Best practice: use ENTRYPOINT in exec form for the main process, and CMD for default arguments. Avoid shell form — exec form doesn't spawn a shell and properly handles signals.

3. What is a multi-stage build?

A multi-stage build uses multiple FROM statements in a single Dockerfile. Each stage can produce intermediate artifacts, and the final stage only copies what's needed — eliminating compilers, build tools, and dev dependencies from the production image. 

# Stage 1: Build
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY . .
RUN go build -o server .

# Stage 2: Runtime (only copies the binary)
FROM alpine:3.19
WORKDIR /app
COPY --from=builder /app/server .
EXPOSE 8080
ENTRYPOINT ["./server"]

The result is a tiny production image (often <20 MB instead of >800 MB) with a smaller attack surface.

4. What is the difference between COPY and ADD?

Both instructions copy files into the image, but ADD has extra capabilities: it can automatically extract local .tar archives into the destination path, and it can fetch files from remote URLs. 

# COPY — straightforward file copy (preferred)
COPY ./config /app/config

# ADD — auto-extracts archives, fetches URLs (use only when needed)
ADD app.tar.gz /app/

Prefer COPY for predictability. Only use ADD when you specifically need archive extraction — its implicit behaviour can lead to unexpected results in builds.

5. What are Docker networking modes?

Docker supports several network drivers:

  • bridge (default) — containers get their own IP on a virtual bridge network; containers can communicate with each other if on the same bridge network.
  • host — container shares the host's network stack; no port mapping needed, but no isolation.
  • none — completely isolated; the container has no network interfaces.
  • overlay — spans multiple Docker hosts (Docker Swarm / Kubernetes); enables cross-host container communication.
# Inspect a container's network
docker inspect <container> --format '{{json .NetworkSettings.Networks}}'

# Connect a running container to a second network
docker network connect my-network my-container

6. What is a Docker volume?

A volume is a mechanism to persist data generated by and used by containers. Unlike bind mounts (which link a host path), volumes are managed by Docker and stored at /var/lib/docker/volumes/. 

# Create a named volume
docker volume create pgdata

# Mount it into a container
docker run -v pgdata:/var/lib/postgresql/data postgres:16

# List volumes
docker volume ls

# Remove unused volumes
docker volume prune

Volumes survive container deletion, can be shared between containers, and are the recommended way to handle persistent data (databases, file uploads) in Docker.

7. How do you reduce Docker image size?

Key strategies:

  • Use a minimal base: alpine, distroless, or scratch
  • Use multi-stage builds to exclude build tools from the final image
  • Combine RUN commands to reduce the number of layers
  • Clean up in the same RUN layer where packages are installed
  • Use a .dockerignore file to exclude node_modules, .git, tests, and docs
# Combine and clean in one layer
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*

8. What is Docker Compose used for?

Docker Compose lets you define multi-container applications in a single docker-compose.yml file, then start the entire stack with one command. It handles networking, volumes, environment variables, and service dependencies declaratively. 

services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=db
    depends_on:
      - db

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: secret
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
docker compose up -d      # Start all services detached
docker compose logs -f    # Tail all logs
docker compose down -v    # Stop and remove containers + volumes

9. What is Docker Hub and how do registries work?

Docker Hub is the default public registry for Docker images — it's where docker pull nginx fetches from. A registry stores and distributes versioned image layers identified by name and tag (e.g. nginx:1.25-alpine). Private registries (AWS ECR, GCR, Harbor) work the same way but require authentication. 

# Tag and push to Docker Hub
docker tag myapp:latest username/myapp:1.0
docker push username/myapp:1.0

# Tag and push to AWS ECR
docker tag myapp:latest 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:1.0
docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/myapp:1.0

10. What is the difference between a Docker image and a container?

An image is a read-only, layered template that contains the application code, runtime, libraries, and configuration. It's built from a Dockerfile and stored in a registry. Think of it as a class definition.

A container is a running instance of an image. When you run a container, Docker adds a thin writable layer on top of the image's read-only layers. Multiple containers can run from the same image simultaneously, each with its own writable layer. Think of it as an object instance. 

# Images
docker images                    # List local images
docker pull nginx:alpine         # Pull from registry
docker build -t myapp:1.0 .     # Build from Dockerfile
docker rmi myapp:1.0            # Remove image

# Containers
docker run -d -p 80:80 nginx    # Create + start container
docker ps                        # List running containers
docker stop <id>                # Stop container
docker rm <id>                  # Remove container
